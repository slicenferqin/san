GBrain 对 San 的启发

这份笔记总结 garrytan/gbrain 的核心架构思想，并把它转译成 San 基于 Oh My Pi 继续定制时可以采用的路线。结论很明确：San 应该先做薄 harness，而不是重新实现一个完整的 brain 系统。

一句话结论

GBrain 不是聊天 agent harness，而是独立的长期知识脑。 对 San 来说，最佳借鉴方式不是把 GBrain 代码塞进来，而是在 San 里实现 brain-first、memory router、push context 和 fat skills 的工作流层，然后通过 MCP 外接 GBrain 或其他 brain 后端。
San 的定位 薄 harness：负责上下文路由、工具调度、默认行为和 agent 体验。
Mnemopi 的定位 本地 operational memory：偏好、配置、工作习惯、项目级记忆。
GBrain 的定位 外部 knowledge brain：人、公司、项目、会议、原创想法、资料库。

GBrain 做了什么

GBrain 的 README 把自己定义为 agent 缺失的 brain layer：它提供混合检索、知识图谱、 合成回答、gap analysis、MCP 接入、schema packs、后台 dream cycle 和 job queue。 它可以本地跑 PGLite，也可以用 Postgres + pgvector 做大规模或团队共享。

它最重要的产品区别不是“能搜”，而是“能综合”。普通 RAG 返回一批 chunk，GBrain 试图返回 带引用的答案，并指出当前知识的缺口、陈旧信息和矛盾。

能力 GBrain 的做法 San 的借鉴方式
长期知识 Markdown repo + DB index，brain repo 是 system of record。 先不内置完整 brain；保留接入外部 brain 的接口。
检索 关键词、向量、BM25、RRF、graph signals、reranker。 San 先实现调用策略：先查本地/brain，再上外部搜索。
知识结构 schema packs 定义 page types、link types 和文件归档规则。 San 可以先做轻量 taxonomy/resolver，后面再 schema 化。
Agent 接入 MCP 工具面，支持 Codex、Claude Code、Cursor 等。 San 可把 GBrain 当 MCP server，而不是复制它。
后台循环 dream cycle、cron、Minions job queue。 San 后续做小型 post-turn consolidation，再考虑 daemon。

最关键的边界：Brain vs Memory vs Session

GBrain 的一个好判断是把上下文分层。这个边界如果不清楚，agent 会把人脉资料塞进偏好记忆， 又把用户回复风格写进知识库，最后检索质量和行为稳定性都会变差。

层 存什么 San 中建议的落点
Session context 当前任务、刚刚说过的话、临时决策。 对话窗口、todo、task state。
Agent memory 用户偏好、工具配置、工作习惯、项目操作约定。 mnemopi 或 OMP memory backend。
Brain 关于世界的知识：人、公司、项目、会议、想法、资料。 外接 GBrain，或以后做 San brain adapter。
不要混层。 “用户喜欢简洁回答”应进 agent memory；“Alice 是 Acme 的工程负责人”应进 brain； “这轮正在修哪个 bug”应留在 session。

对 San 最有价值的四个设计

1. Brain-first lookup

在调用 web search、GitHub、LinkedIn、数据库或其他外部 API 前，先查已有记忆和 brain。 这能减少重复检索，也能让回答带上个人上下文，而不是给用户一个公开网页摘要。

2. Push-based context

不等模型想起来才 recall，而是每轮对话前从最近几轮里抽实体和主题，自动推送少量高置信记忆。 对 San 来说，这可以成为比“手动 recall”更明显的体验升级。

3. Compiled truth + timeline

每个知识页分两层：上半部分是当前综合判断，下半部分是 append-only 时间线。 这比单纯存碎片更适合长期演化的想法、人脉和项目。

4. Thin harness, fat skills

San 不需要把所有行为写进 TypeScript。更适合的路线是：核心 harness 保持小而稳定， 把判断流程写进 markdown skill，把触发条件写进 resolver。

不建议现在复制的部分

  • 不要直接 vendor GBrain。它包含 PGLite/Postgres、pgvector、OAuth MCP、schema packs、job queue、eval、dream cycle，短期会把 San 的复杂度拉爆。
  • 不要马上重做完整 schema pack。先做轻量分类规则和归档习惯，等真实使用数据出来再结构化。
  • 不要一开始做后台 daemon。先把每轮对话前后的小循环跑稳，再扩展成 cron/dream cycle。
  • 不要把 mnemopi 当成 GBrain 的替代品。mnemopi 更适合 agent 操作记忆，GBrain 更适合外部世界知识。

建议路线

第一阶段：San memory loop

开启 mnemopi，实现“每轮前自动 recall、每轮后候选 retain”的轻量循环。 目标是让 San 先拥有稳定的工作偏好和项目记忆。

第二阶段：San resolver + skills

增加 SAN.md、skills/RESOLVER.md 和几份基础 skill： brain-first、memory routing、idea capture、research ingest、project decision log。

第三阶段：GBrain MCP adapter

让 San 可选连接 gbrain serve，先走本地 stdio MCP，后续再支持远程 HTTP MCP。 San 只负责调用策略，不拥有 GBrain 的数据库细节。

第四阶段：Compiled truth 工作流

给想法、项目、人和资料建立“当前综合判断 + 时间线”的写入习惯。 这一步可以先用 markdown skill 驱动，不必一开始写成硬代码。

第五阶段：Dream cycle 的 San 版本

当前四步跑稳后，再做轻量后台整理：合并重复记忆、提炼项目决策、生成次日上下文、 发现未归档资料和陈旧判断。

可以转成任务的候选项

  1. 打开 San 的 memory.backend=mnemopi，并明确 scoping 策略。
  2. 新增一份 San 自己的 agent operating protocol，先放在 .san/ 或 docs/san/。
  3. 实现 memory router：把“该记哪里”的判断变成可测试的规则。
  4. 做 pre-turn recall：从用户最新消息和最近几轮提取关键词，自动查 mnemopi。
  5. 做 post-turn retain candidate：把候选记忆展示/记录，不直接静默污染 memory。
  6. 验证 San 是否能通过 MCP 连接本地 GBrain：gbrain init --pglite + gbrain serve。
