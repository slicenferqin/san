Agent Loop、Amp 与 San 的落地方向

这份笔记把本地 tabbit 调研、Amp 公开手册和 San 当前基于 Oh My Pi 的能力对齐， 目标是提炼一组可以在 San 中逐步落地的 loop 工程方案。

一句话结论

San 不应该只是一个更会聊天的 agent，而应该成为一个更会组织 agent 工作的 agent。 也就是：用 Oh My Pi 做执行底座，用 Amp 式多线程和角色体验改善使用方式，用 GBrain 式长期脑处理知识， 再用 San 自己的 loop discipline 把任务拆解、执行、验收、复盘和记忆串起来。
核心抽象 从 prompt 转向 loop：优化的不再是一句话，而是行动、观察、评估、重试、终止的控制结构。
核心组织 Commander、Worker、Supervisor 分离，尤其是 supervisor 要有独立上下文、独立模型和门禁权。
核心资产 每次 loop 的计划、产出、验收报告、失败原因和复盘，都应该沉淀为可检索、可复用的项目资产。

为什么 Agent Loop 比 Prompt 更关键

Prompt 仍然重要，但它已经从“产品本身”降级成 loop 的一个组件。一个能工作的 agent 系统， 真正的上限来自它如何持续选择动作、读取观察、更新状态、处理错误、决定是否停止。

1 Act

调用工具、执行代码、写文件、派发子任务。

2 Observe

读取 stdout、diff、测试结果、错误码和外部平台数据。

3 Judge

对照目标、验收标准和预算判断是否有进展。

4 Repeat / Finish

继续、打回、升级给人，或显式结束。

工程问题 Prompt 做不到的点 San 应该落地的机制
终止 靠模型“自觉停”很不可靠。 显式 yield/finish、最大轮数、预算上限、无进展检测。
上下文 长上下文会腐烂，注意力会稀释。 子任务隔离、压缩、结构化任务账本、memory/brain recall。
错误恢复 自然语言错误很难稳定驱动下一步。 结构化错误和结构化验收报告。
可观测性 没有轨迹就无法调试 loop。 保存 plan、tool calls、diff、report、cost、retry reason。
复盘 一次任务成功不等于系统变聪明。 把失败模式写入 checks、skills、memory 或 brain。

三角色闭环

本地 tabbit 文档里最值得保留的判断是：executor 和 looper 不应该是同一个 agent； 干活、监督、指挥最好分离，甚至使用不同模型。原因不是形式主义，而是上下文隔离、权限隔离和失败模式隔离。

Commander

负责需求分析、任务拆解、读取报告、决定下一轮动作。

工具权限应该偏只读和派发，避免它亲自改代码导致职责混乱。

Worker

负责执行一个边界明确的子任务。

可以拥有 edit/write/bash，但不应该能继续无限派发任务。

Supervisor

负责写验收标准、跑测试、做 code review、产出门禁报告。

最好使用不同 provider/model，并且不能直接修改代码。

角色 上下文 工具边界 模型策略
Commander 全局目标、任务 DAG、历史验收报告、复盘。 Read、Task、Todo，尽量无 Edit/Write/Bash。 强模型，少调用，重决策。
Worker 当前子任务、相关文件、验收标准。 Edit、Write、Bash、Read、Grep，限制 Task。 中低成本模型，多调用，重执行。
Supervisor 任务说明、验收标准、diff、测试输出。 Read、Grep、Bash，禁止 Edit/Write。 异构模型，重判断。

Amp 可借鉴的点

Amp 的公开手册和新闻里有几个很直接的产品/工程启发：multi-model、threads、subagents、oracle、 librarian、checks、skills 懒加载、MCP-in-skill、plugin API、durable agent loop 和 remote control。

Amp 设计 价值 San 中的对应方案
deep / smart / rush 按任务复杂度控制模型、token 和检查强度。 San modes：rush、smart、deep。
Threads 一个任务一个线程，可搜索、可引用、可分享。 San run ledger：一个 loop 一个 run id，保存计划和报告。
Subagents 隔离上下文、并行探索、减少主线程污染。 使用 OMP task agents 作为 Worker/Supervisor。
Oracle 复杂推理时调用更强的第二意见模型。 San oracle：只在 deep 或 supervisor 不确定时调用。
Librarian 远程代码库和外部源码研究。 San research/librarian skill，绑定 GitHub/GBrain/web MCP。
Checks 把团队隐性规则变成 review 子代理。 .omp/checks/*.md，每个 check 生成一个 supervisor 子任务。
MCP in skills 工具只在需要时出现，降低上下文和工具污染。 San skill-bound MCP：research/browser/brain/ops 分别加载工具。

Oh My Pi 已经给了什么

San 不是从零开始。当前仓库已经有 task agent discovery、skills、hooks、extensions、MCP、memory、 compaction、yield、session tree 等能力。第一阶段应该尽量组合这些能力，而不是先改 agent loop 核心。

OMP 能力 当前位置 San 用法
Task agents .omp/agents、docs/task-agent-discovery.md 定义 commander、worker、supervisor、oracle、librarian。
Agent output schema agent frontmatter output、SDK outputSchema 强制 worker/supervisor 输出结构化结果。
Yield tool SDK requireYieldTool 让子代理显式结束并交付报告。
Hooks / extensions .omp/extensions、tool_call、tool_result 实现权限门禁、预算上限、危险命令拦截。
Skills .omp/skills/*/SKILL.md 把 San loop 工作流写成可懒加载的能力包。
Mnemopi memory.backend=mnemopi 保存项目操作记忆、失败模式、用户偏好。

第一阶段落地方案

新增 San 三角色 agents

在 .omp/agents/ 下新增 san-commander.md、 san-worker.md、san-supervisor.md。 先通过 prompt、工具白名单和 output schema 分离职责。

新增 San Checks 规范

建立 .omp/checks/ 目录。每个 markdown check 描述一个可审查规则： 安全、性能、边界条件、产品体验、项目约定。后续由 supervisor 子代理逐个执行。

新增 san-loop 命令

在 .omp/commands/san-loop.md 中定义闭环工作流： commander 拆解、worker 执行、supervisor 验收、commander 复盘并决定是否 retry。

定义结构化报告

统一 WorkerResult 和 ReviewReport： verdict、changed_files、tests_run、 defects、retryable、next_action。

打开项目记忆

使用 mnemopi 保存 loop 复盘：反复失败的模式、有效 checks、用户偏好、 项目特殊约定。先做人可见的 retain，不做静默污染。

再做代码级 orchestrator

当前四步跑稳后，再考虑用 SDK 写真正的 San orchestrator，获得更强的状态机、预算统计、 并发任务和远程控制能力。

San Modes 草案

模式 适合任务 默认 loop
rush 小改动、机械任务、明确修复。 单 worker，轻量检查，低成本模型。
smart 默认开发任务、中等复杂度需求。 commander → worker → supervisor → commander。
deep 架构改动、难 bug、高风险修改。 强 commander、异构 supervisor、oracle 二审、brain/memory recall。

必须装的护栏

  • 同一子任务最多重试 N 轮，超过后升级给用户。
  • 连续两轮 diff 无实质变化时停止循环。
  • 预算和 token 上限必须对 commander 可见。
  • 危险 bash 命令必须走 hook 或 supervisor gate。
  • Supervisor 的结论必须结构化，不能只写自然语言感想。
  • 能用确定性测试解决的，不交给 LLM judge。
循环不是越多越好。 不可验证、低价值、一次性的任务不应该进入复杂闭环。San 的目标是把可重复、可验收、高价值任务做成复利系统。
