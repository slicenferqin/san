San v0.1 Review · 2026-06-29

上下文稳态规划方向正确，但落地要把“摘要账本”和“上下文注入”当成两个不同系统

本文基于 docs/research 的五份研究文档、当前 packages/coding-agent 代码路径、CodeGraph 查询以及 Amp/GBrain 当前公开资料，对 San v0.1 计划做可行性审阅和落地风险拆解。

直接结论

Verdict
按当前大方向推进，可以达成 San v0.1 的目标。 选择 Context Steady State 作为 v0.1 主线是正确的：它足够差异化，也能承接后续 Agent Loop 和 GBrain/brain 路线。M1 用 CustomEntry 保存 TurnDigest 的方案也与现有 session 设计匹配。
需要修正的是落地口径：M1 不是“监听任意 agent_end 就写摘要”，而是要接在 AgentSession 已经判定本轮真正 settled 的维护路径之后；M2 也不是“把 digest 简单塞进 prelude messages”，而是要明确注入消息的生命周期、token 统计、append-only cache 影响和 debug view。

读完规划后的目标理解

五份文档实际形成了一个清晰的路线：

  1. san-v0.1-plan-evaluation.html 把 v0.1 锁定为 Context Steady State，不把 v0.1 降级成“摘要 MVP”。
  2. context-steady-state-san.html 定义完整目标：每轮结束沉淀 TurnDigest，下一轮前组装 ContextPacket，并把上下文稳定控制在质量窗口内。
  3. context-steady-state-m1-technical-plan.html 把第一个里程碑收窄为 post-turn TurnDigest 生成与持久化。
  4. agent-loop-amp-san.html 把 Commander/Worker/Supervisor、Checks、Oracle 等执行闭环放到 v0.2 更合理。
  5. gbrain-san-ideas.html 把 brain/memory/session 分层放到 v0.3+，避免 v0.1 过早外接完整 brain。

这个排序是对的。先做上下文稳态，再做执行闭环，再做 memory/brain 路由，能避免一开始把 prompt、agent orchestration、长期知识库和 session state 搅在一起。

规划正确的地方

v0.1 目标没有选偏 Context Steady State 是底层能力，不是表层 workflow。它能让后续三角色、checks、brain recall 都消费同一套受控上下文。
M1 范围拆得合理 只做 digest ledger，不改 core agent loop，不注入下一轮 prompt，能先建立数据来源和调试面。
CustomEntry 是合适载体 当前类型注释明确说 CustomEntry 不进入 LLM context，而 CustomMessageEntry 才进入上下文。这正好符合 M1 的“先存不注入”。
外脑没有提前引入 GBrain 当前公开定位就是 brain layer，提供 synthesis、graph traversal、gap analysis、MCP 等。v0.1 直接接它会明显扩大复杂度。
Amp 启发放到 v0.2 合理 Amp Owner's Manual 当前入口仍是产品/手册形态；subagent、checks、oracle、librarian 类能力适合作为 San 执行闭环，而不是上下文底座。
质量窗口思路正确 不把模型标称窗口当作工作目标，而是把 1M 当缓冲，把可控上下文控制在稳定窗口内，符合长期 agent 使用经验。

主要问题

问题 为什么会影响目标 建议修正
M1 文档把 agent_end 说得过于简单 当前 AgentSession 的 agent_end 不是单一事件点。它会经过 empty stop、unexpected stop、retry、compaction、todo continuation、session_stop continuation 等维护路由。 Digest 生成要挂在“最终 settled turn”的内部维护路径上，并且只对真正终止的用户轮次写入。不要用普通 async subscriber 在外面监听所有 agent_end。
“每轮”边界需要更精确定义 一个用户 prompt 内可能有多次 model call、tool-use turn、auto-retry、auto-continue、session_stop continuation。直接按底层 agent_end 或 turn_end 计数会重复或漏写。 M1 先定义 source.fromEntryId / source.toEntryId / promptGeneration / 用户消息 entry id 的组合，并用它做去重和可追溯范围。
M2 的“prelude messages”说法不够 #promptWithMessage() 里已经有 plan/goal/eager todo/eager task/keyword/image/custom hook/file mention/Mnemopi 相关消息。新增 ContextPacket 如果只插在 prependMessages，容易和现有动态上下文排序、token 统计、pre-prompt compaction、append-only context 产生副作用。 给 ContextPacket 单独建 builder 和 debug record，明确排序：稳定/半稳定内容尽量走 system prompt 或低频 checkpoint；动态 digest/retrieval 放在 current prompt 前，但必须参与预算和 debug view。
ContextPacket debug view 还没有技术落点 v0.1 的验收依赖“可解释地复用”。如果只有 session entry 和隐式 prompt 注入，用户无法判断模型到底看到了什么。 M2 前先规划 san.context_packet custom entry 或 debug-only transcript marker，记录 layers、entry refs、token estimate、trim decisions 和 final injected message ids。
预算稳态不能只依赖本地估算 现有代码已经把 assistant message 的 contextSnapshot 当作 provider-anchored usage。只用 estimateTokens() 可能无法证明多轮后真实请求稳定。 v0.1 验收同时记录 local estimate 和 provider/contextSnapshot 两套数据。debug view 要显示差值，而不是只显示一个估算数字。
memoryCandidates 的边界容易越界 M1 输出 memoryCandidates 是合理的，但如果后续直接 auto-retain，很容易把 session 临时状态、项目事实、用户偏好和 world knowledge 混层。 M1 只产候选，不写长期 memory。v0.3 前保留人工/显式策略。分类上沿用 GBrain 的 brain vs memory vs session 边界。

按规划能否达成目标

可以，但要把 v0.1 拆成四个明确闭环，而不是只按“Phase 1/2/3”粗分：

闭环 必要交付 验收口径
M1: Digest Ledger san.turn_digest custom entries、list/read helper、fallback digest、去重、失败不阻断。 真实用户轮次结束后能读到 digest；resume 后仍能读；不进入 LLM context。
M2: ContextPacket Injection 读取 digest ledger，组装最小 ContextPacket，注入下一轮，写 debug entry。 下一轮 payload 或 debug view 能证明上一轮 digest 被实际消费。
M3: Budget Steady State qualityWindowTokens、layer budget、trim decisions、provider/contextSnapshot 对照。 构造 10-20 轮噪声会话后，下一轮 prompt 仍稳定在质量窗口内，裁剪日志可解释。
M4: Recall Boundary 轻量 Mnemopi/historical recall 接入，只读或候选模式，不做长期静默写入。 跨轮/跨会话能召回相关偏好或项目决策，但不会把 session 临时信息污染长期 memory。
v0.1 发布口径应该是 M1-M3 必须完成，M4 可以作为受控增强。只有 M1 的话只是“摘要账本”，还不是上下文稳态。

最容易走偏的地方

  1. 把摘要当成上下文稳态本身。 Digest 是原料，不是目标。目标是下一轮实际消费、可解释、可裁剪、预算稳定。
  2. 把所有 digest 都塞进下一轮。 这样只是把原始 transcript 换成摘要 transcript，长期仍会膨胀。必须有最近 N 条、checkpoint、相关召回和预算裁剪。
  3. 让 LLM 生成的 digest 覆盖证据。 Digest 只能存短摘、路径、entry id、hash、行号。文件内容和工具输出要按需重读，不能信摘要替代证据。
  4. 在错误/中断/自动续跑时重复写 digest。 aborted、retryable error、empty stop、session_stop continuation 都会制造边界错觉。必须用 source range 去重。
  5. 污染 append-only cache 前缀。 动态 ContextPacket 如果放在太靠前的位置或每轮重写稳定前缀，会破坏 prompt cache。稳定规则和低频 checkpoint 放前，动态召回靠后。
  6. 把 debug UI 做成产品噪声。 v0.1 必须可调试，但默认用户体验不要被内部 markers 淹没。建议用命令/debug view 或隐藏 custom entry 展示。
  7. 过早接 GBrain 或三角色。 它们都很诱人，但会把当前目标从 context system 拉成 orchestration/knowledge system。v0.1 先把底座跑稳。

落地护栏

护栏 具体做法
先做 deterministic fallback M1 先保证不用 LLM 也能写出 schema 完整的 fallback digest，再接 summary model。这样测试和持久化 contract 稳定。
summary LLM 只能是 side request 不要把 digest 生成走主 agent loop，不要触发工具、session message 或新的 agent_end。用独立 helper 调 completeSimple/instrumentedCompleteSimple。
所有 digest job 绑定 generation 和 source range 创建任务时记录当前 session id、leaf id、from/to entry、prompt generation。写入前二次确认 session 未切换、未 abort、source range 仍在当前 branch。
失败默认不影响用户 turn LLM 摘要失败、JSON parse 失败、timeout 只写 fallback 或 logger warning；不能让 prompt() promise 失败。
M2 先只读最近 digest 第一版只注入最近 1-3 条 digest 加当前 prompt，先证明“下一轮实际复用”。不要同时做 checkpoint、recall、brain、subagent。
debug entry 与注入 entry 分离 ContextPacket debug 可以持久化为 san.context_packet custom entry；实际注入可以是隐藏 custom message 或 transformContext 结果。两者要能互相引用。
验收用真实 session，不只测 helper 测试必须跑过 AgentSession.prompt() 或现有 message pipeline，验证 session entries、resume、buildSessionContext、下一轮 payload/ContextPacket debug。

建议的实现顺序

  1. 新增 packages/coding-agent/src/context-steady/types.ts 和纯函数 helpers：schema normalize、source range、fallback digest、list/filter digest entries。
  2. 加设置项，但默认先可控。建议先做 san.contextSteady.enabled、san.contextSteady.digest.enabled、san.contextSteady.digest.persistFallback、san.contextSteady.digest.timeoutMs。
  3. 在 AgentSession 内部 settled agent_end 维护尾部接入 digest service。不要走外部 subscriber 作为主路径。
  4. 先写 fallback digest 并持久化；测试通过后再接 LLM digest prompt。Prompt 必须放 .md，用 Handlebars 渲染。
  5. M2 单独写 ContextPacket builder，先从 digest ledger 读最近 1-3 条，写 debug custom entry，再注入下一轮。
  6. 加多轮 fixture/e2e，证明 digest 被消费，且上下文 token 不随轮数线性增长。

外部背景

  • Amp Owner's Manual：当前公开入口支持把 Amp 作为 agent 产品/手册参考；其 subagents、checks、oracle、librarian 等概念更适合 San v0.2 执行闭环。
  • GBrain README：GBrain 明确定位为 agent brain layer，强调 synthesis、graph traversal、gap analysis、MCP 接入和 dream cycle。它适合外部 brain，不适合塞进 v0.1。
  • GBrain Brain vs Memory vs Session：其分层规则支持 San 规划里的“不混层”判断：world knowledge 进 brain，operations/preferences 进 agent memory，当前任务留 session。

最终建议

不建议改 v0.1 目标。应该继续以“上下文稳态”为主线推进，但把规划文档里的 M1 接入点更新成更严格的 settled-turn contract，并在 M2 前补一份 ContextPacket 技术方案，重点写清楚 layer 顺序、注入位置、debug entry、token 预算和 append-only cache 策略。

最小可交付路径是：M1 先做可靠 digest ledger；M2 做可见、可验证的下一轮复用；M3 做质量窗口和裁剪日志。走完这三步，San v0.1 才真正达成“每轮上下文能稳定收敛”的目标。
