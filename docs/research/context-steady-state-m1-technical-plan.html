San Context Steady State · Milestone 1

第一个里程碑：Post-turn TurnDigest 生成与持久化

v0.1 的最终目标仍然是上下文稳态。M1 只解决第一块：每轮 agent 停下后，San 能把本轮工作沉淀成结构化 TurnDigest，并可靠写入 session。后续 M2 才开始让下一轮 ContextPacket 消费这些 digest。

结论

M1 不改 core agent loop，不新增 session entry 类型。 在 packages/coding-agent 内新增 context-steady 模块，监听 agent_end 后生成 TurnDigest，用现有 sessionManager.appendCustomEntry("san.turn_digest", data) 持久化。这样最小化迁移风险，并为 M2 的 ContextPacket 组装提供稳定数据源。

M1 目标与非目标

目标 1 每轮 agent_end 后自动生成一个结构化 TurnDigest。
目标 2 Digest 持久化到当前 session JSONL，可 resume 后读取。
目标 3 失败不影响主 agent 结果，只记录 warning/debug。
非目标 M1 不把 digest 注入下一轮 prompt。那是 M2 的 ContextPacket 工作。
非目标 M1 不做复杂 memory router，不写入长期 Mnemopi，最多产出 memoryCandidates。
非目标 M1 不改变 compaction、append-only context、core agent-loop 语义。
M1 是 v0.1 的第一个里程碑，不是 v0.1 的完整目标。v0.1 发布验收仍然必须达到上下文稳态。

为什么用 CustomEntry

当前 session 系统已经有 CustomEntry，用途就是给扩展或系统状态持久化自定义数据。它不会进入 LLM context，正好适合 M1：digest 先存下来，不影响现有上下文构建。 

{
  "type": "custom",
  "customType": "san.turn_digest",
  "data": {
    "schemaVersion": 1,
    "turnId": "entry-id-or-generated-id",
    "source": {
      "sessionId": "...",
      "fromEntryId": "...",
      "toEntryId": "..."
    },
    "digest": { "...": "..." }
  },
  "id": "...",
  "parentId": "...",
  "timestamp": "..."
}
后续如果 digest 成为一等 session 概念，再考虑新增正式 SessionEntry 类型和 migration。M1 先不要引入这类迁移成本。

TurnDigest 数据结构

export interface TurnDigest {
  schemaVersion: 1;
  turnId: string;
  sessionId: string;
  createdAt: string;
  model?: string;
  userIntent: string;
  actionsTaken: string[];
  decisions: string[];
  filesTouched: Array<{
    path: string;
    action: "read" | "modified" | "created" | "deleted" | "unknown";
    reason?: string;
  }>;
  toolEvidence: Array<{
    tool: string;
    summary: string;
    entryIds?: string[];
  }>;
  factsLearned: string[];
  openQuestions: string[];
  risks: string[];
  nextSteps: string[];
  memoryCandidates: Array<{
    content: string;
    type: "preference" | "project_fact" | "decision" | "workflow" | "other";
    importance: number;
  }>;
  tokenStats?: {
    input?: number;
    output?: number;
    cacheRead?: number;
    cacheWrite?: number;
    total?: number;
  };
  fallback: boolean;
}

字段设计原则：摘要保留“决策和证据索引”，不塞完整工具输出。文件内容要靠路径、entry id、后续 read/codegraph 重读保证新鲜。

事件时序

1 message_end

AgentSession 已按现有逻辑持久化用户、assistant、toolResult 等消息。

2 agent_end

等现有 agent_end maintenance 路径完成 pending bash/python/irc flush。

3 collect turn

从 session branch 中定位本轮起止 entry，提取最近新增消息和工具证据。

4 summarize

优先调用配置的 summary model；失败则 deterministic fallback。

5 persist

appendCustomEntry("san.turn_digest", data) 写入 session。

触发点要放在 settled agent_end 后，而不是每个 message_end。否则工具结果尚未全部落盘，digest 会漏证据。

摘要生成策略

路径 行为 用途
LLM digest 用 summaryModelRole 解析模型，调用 instrumentedCompleteSimple，输出 JSON。 默认路径，质量最好，能提炼决策和风险。
Schema normalize 解析 JSON，缺字段补空数组，字符串长度和数组长度做上限。 防止模型输出污染 session。
Fallback digest LLM 失败、超时、无模型时，从用户最后 prompt、assistant 文本、工具名生成保守摘要。 保证 M1 的持久化 contract 稳定。
Skip digest 没有真实用户/assistant 消息、synthetic housekeeping、已存在同 turn digest。 避免空会话和重复写入。

san.contextSteady.digest.enabled = true
san.contextSteady.digest.maxInputMessages = 40

san.contextSteady.digest.maxTokens = 2000

输入裁剪

M1 的 digest 不是要总结整段会话，而是总结“本轮”。输入给摘要模型时按下面规则裁剪，避免摘要本身变成上下文膨胀源。

内容 保留规则 上限建议
用户 prompt 完整保留本轮用户 prompt；如果极长，保留头尾并标记截断。 16K chars
assistant 文本 保留关键回答、计划、结论；丢弃重复思考和长解释。 24K chars
tool result 只保留工具名、状态、短摘要、文件路径、entry id。 20 条
文件变更 从 edit/write/apply_patch/bash git diff 线索提取路径和动作。 50 个路径
token usage 汇总本轮 assistant usage。 结构化数字

失败与并发处理

  • 失败不阻断：digest 生成失败不能影响用户看到的 agent 结果，也不能让 prompt promise 失败。
  • 去重：以 source.fromEntryId + source.toEntryId 或 turnId 去重，避免 resume/auto-continue 重复写。
  • 超时：LLM digest 超过 timeoutMs 后取消，写 fallback digest 或跳过。
  • 队列：同一 session 同时只允许一个 digest job，后续 agent_end 串行排队。
  • dispose：session dispose 时等待短时间 drain；超时放弃，不阻塞退出。
  • 日志：用 logger.debug/warn，不碰 console。

验收标准

  1. 开启 san.contextSteady.enabled 后，任意真实用户轮次结束都会生成一条 san.turn_digest custom entry。
  2. Digest 包含 userIntent、actionsTaken、decisions、filesTouched、nextSteps 等核心字段。
  3. LLM 摘要失败时，主 agent 结果不失败，session 中可选择写入 fallback digest。
  4. Resume 同一 session 后能读取历史 digest。
  5. buildSessionContext() 不把 digest 自动注入 LLM 上下文。
  6. 新增测试可通过 bun check 和 package-local test。

后续承接

M1 完成后，M2 的技术方案就可以围绕 ContextPacket 展开：读取 san.turn_digest ledger，选择最近 N 条和 checkpoint summary，插入到下一轮 AgentSession.prompt() 的 prelude messages，并开始验证 qualityWindowTokens。

M1 的关键价值不是“多一个摘要文件”，而是建立 San 上下文稳态的数据账本。没有这个账本，M2 无法稳定组装 ContextPacket。
