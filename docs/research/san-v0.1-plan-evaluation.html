San v0.1 Decision

v0.1 目标锁定“上下文稳态”，范围只做里程碑拆分

本文对三份研究方案做 v0.1 适配评估：GBrain 启发、Agent Loop/Amp 启发、上下文稳态。结论是：上下文稳态就是 San v0.1 的最终目标；Agent Loop 方案适合作为后续执行闭环增强；GBrain 方案适合作为后续 memory/brain 外接路线。

一句话结论

San v0.1 的最终目标就是 Context Steady State： 每轮 agent_end 后生成 TurnDigest，下一轮 prompt 前组装可调试的 ContextPacket，默认控制在黄金上下文窗口内，并能证明上下文在多轮后保持稳定。可以按内部里程碑推进，但不改变 v0.1 的最终目标。

评估标准

标准 为什么重要 v0.1 判断口径
闭环完整性 v0.1 必须能让用户实际感受到 San 变好了。 一次会话内能观察到输入、处理、状态沉淀、下一轮复用。
改动半径 fork 初期最怕一上来改 core loop，风险高且难 debug。 优先在 packages/coding-agent session 层落地。
可验证性 v0.1 要能明确判断做成没做成。 能通过 debug view、session entry、token 预算、下一轮上下文确认。
差异化 San 不只是 OMP 改名，v0.1 要有一个明确方向。 用户能感知“San 会管理上下文”，而不只是多几个 prompt。
后续承接 v0.1 不应是一次性实验，要能接 v0.2/v0.3。 能自然接入 memory、subagent、checks、brain adapter。

方案评分

方案 闭环 改动半径 可验证 差异化 v0.1 适配结论
上下文稳态 9/10 7/10 9/10 9/10 最佳 v0.1 主线。目标不变，实施上拆成里程碑推进。

排序

1

上下文稳态：v0.1 主线

它最贴近 San 的底层差异化：不是“多一个 agent loop”，而是让 agent 在长任务中持续保持可控上下文。这个方向可以直接利用现有 AgentSession.prompt()、agent_end event、Mnemopi 和 compaction，不需要先重写 core loop。

v0.1 的最终验收不能降级为两个能力；TurnDigest 和 ContextPacket debug view 只是最早的内部里程碑。最终仍要证明“San 能沉淀每轮、影响下一轮，并在多轮之后把上下文稳定控制在黄金窗口内”。

2

Agent Loop / Amp：v0.2 执行闭环

三角色、San Checks、yield contract 很有价值，但更像“任务执行质量提升”。如果作为 v0.1，容易变成一组 agents/commands，而不是 San 的核心能力。它应该建立在上下文稳态之上：Commander、Worker、Supervisor 都应该消费同一个 ContextPacket。

3

GBrain 启发：v0.3+ 记忆和外脑

GBrain 的 brain-first、compiled truth、push context 是长期方向，但 v0.1 不应引入外部 brain 复杂度。当前最合适的吸收方式是只拿“分层思想”：session、memory、brain 不混层。等 San 的 TurnDigest 和 ContextPacket 跑稳，再接 GBrain 或做 brain adapter。

v0.1 最终范围

必须做 每轮结束生成 TurnDigest，记录用户意图、动作、决策、文件、证据、风险、下一步。
必须做 下一轮 prompt 前组装 ContextPacket，至少包含稳定摘要、最近 5 轮、当前 prompt。
必须做 提供 debug view 或 transcript marker，让用户和开发者能看到本轮实际注入了什么上下文。
必须做 多轮后上下文仍稳定在 qualityWindowTokens 内，并有裁剪日志说明删了什么、为什么删。
必须做 下一轮必须实际消费上一轮 digest，而不是只生成摘要文件给人看。
可以做 接入 Mnemopi recall，但只做轻量相关召回，不做复杂 memory router。
暂不做 不做完整三角色 orchestrator，不做 supervisor 自动循环。
暂不做 不接 GBrain，不做后台 dream cycle，不做完整 schema pack。

交付定义

San v0.1 = Context Steady State

1. san.contextSteady.enabled 配置可开关
2. agent_end 后生成 TurnDigest
3. TurnDigest 可持久化并可查看
4. 下一轮 prompt 前组装 ContextPacket
5. ContextPacket 包含：
   - stable summary / digest ledger
   - recent 5 turns
   - current prompt
   - optional mnemopi recall
6. 有 token budget 统计和裁剪日志
7. 多轮后上下文规模稳定在 qualityWindowTokens 内
8. 下一轮实际复用上一轮 digest
9. 不改变 core agent-loop 语义
里程碑可以拆，目标不能拆。 Phase 1 可以只做 digest，Phase 2 可以只做 ContextPacket，但 v0.1 发布口径必须是完整上下文稳态，而不是“摘要 MVP”。

为什么不是三角色作为 v0.1

三角色方案更容易快速做出“看起来很 agentic”的东西，但它的问题是：早期效果高度依赖 prompt，评估也容易漂。没有上下文稳态时，Commander、Worker、Supervisor 共享的仍然是未治理的上下文，循环越多，噪声越多。

三角色应该成为上下文稳态的消费者，而不是 v0.1 的基础层。先把 ContextPacket 做出来，后续每个角色都按自己的职责接收裁剪后的上下文，这样 v0.2 会更稳。

为什么不是 GBrain 作为 v0.1

GBrain 启发很强，但 v0.1 做外脑会引入过多外部系统问题：数据库、MCP server、知识 schema、资料 ingest、引用一致性、后台整理。它很适合作为 San 的长期记忆路线，但不适合作为第一个版本的产品闭环。

v0.1 的目标不是“拥有一个 brain”，而是“每轮上下文能稳定收敛”。外脑要等 San 能清楚地区分 session、memory、brain 后再接。

推荐路线

版本 主题 核心交付
v0.1 上下文稳态 TurnDigest、ContextPacket、多轮预算稳态、下一轮复用、裁剪日志、debug view。
v0.2 执行闭环 Commander / Worker / Supervisor、San Checks、结构化验收。
v0.3 记忆路由 Mnemopi policy、用户偏好、项目决策、post-turn retain candidates。
v0.4 外脑接入 GBrain MCP adapter、brain-first lookup、compiled truth 工作流。

最终建议

选择“上下文稳态”作为 San v0.1，且不要修改这个目标。实现可以分里程碑：先 digest，再 ContextPacket，再预算稳态，再 recall；但 v0.1 的最终判断标准必须是“San 能把每轮工作沉淀成稳定上下文，并在下一轮被可解释地复用，同时多轮后仍控制在黄金窗口内”。
