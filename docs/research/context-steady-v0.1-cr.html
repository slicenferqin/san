二、架构层面——正确的地方

1. 落地口径严格遵守了文档审阅结论

san-v0.1-plan-implementation-review.html 里的护栏基本全部落地：

  • generateDigest 永不 throw，失败只 warn，不阻断用户 turn。
  • LLM digest 是独立 side request，走 completeSimple，不进主 agent loop，不触发新的 agent_end。
  • 所有 digest job 绑定 sessionId + fromEntryId + toEntryId 去重；刻意排除 promptGeneration，因为 resume 后 generation 会重置，entry ID 才是稳定锚点（代码注释有说明）。
  • fallback.ts 不依赖 @oh-my-pi/pi-agent-core，避免了 pi-natives transitive import。

2. #generateTurnDigest 的边界逻辑是对的

文档审阅指出"不要用普通 async subscriber 在外面监听 agent_end"。最终实现在维护路径内部精确接入：

  • session_stop continuation 期间只记录边界，不写 digest。
  • chainEnded 时才产出覆盖完整逻辑 turn 的 digest。
  • abort path 用 boundaryOverride 保全了 continuation 前的原始边界。
  • finally 块负责清理边界状态，即使 LLM 超时或抛出也不留脏状态。

3. ContextPacket trim 顺序正确

buildContextPacket 超预算时：先砍 recall（最 volatile），再从旧到新砍 digest，最后裁剪 checkpoint 尾部字段。这个顺序精确匹配文档对缓存友好度的设计：stable > append-only > volatile。

4. Dogfood 验证了完整流水线

dogfood.ts 的 assertions 覆盖了关键不变量："active context hides injected packet, transcript retains it"，用 buildSessionContext 的两个视图（默认 vs transcript）来验证 CustomMessageEntry display: false 合约，是 M2 注入语义的直接验收。

5. mergeAuthoritativeDigestFields 防止 LLM 篡改系统字段

LLM 生成 digest 后，turnId / sessionId / createdAt / source / tokenStats 全部从 fallback 里强制覆盖，LLM 只负责语义内容（intent、decisions、risks 等）。账本完整性不依赖 LLM 输出合规性。

三、具体问题

按严重度排列。 需要修复 为阻断项； 建议改进 为重要但不阻断； 小问题 为低优先级建议。

需要修复 [1] fallback.ts:218 — token stats total 计算过度累加

const total = input + output + cacheRead + cacheWrite;

cacheRead 和 cacheWrite 是 input 的组成部分（provider 拆分计费）， 不是独立的额外 token。叠加到 input + output 之上会产生虚高的 total。

对照 dogfood.ts 里的测试数据：total = input + output（cacheRead/cacheWrite 均为 0）， 二者不一致。

建议：将 total = input + output，或在 TurnDigestTokenStats 类型注释里明确 total 语义，避免后续消费方做成本统计时误用。

需要修复 [2] digest.ts:289 — JSON fallback 贪婪 regex

const match = /\{[\s\S]*\}/.exec(text);

贪婪匹配会把第一个 { 到最后一个 } 之间所有内容都吃进去。如果 LLM 回复了多个 JSON 对象或正文里有大括号，parse 会失败或抓到错误内容。

工具调用已强制 toolChoice: { type: "tool", name: ... }，fallback 路径触发概率低， 但仍是可靠性风险。建议改为从末尾反向定位最后一个完整 JSON 对象。

建议改进 [3] 四处 as unknown as 双重类型断言

// session.ts, digest.ts, packet.ts, checkpoint.ts 各一处
(sessionManager as unknown as SessionManagerInternal).appendCustomEntry(...)

ReadonlySessionManager 没有暴露 getEntries() 和 appendCustomEntry()， 所有调用者用双重 cast 绕过类型系统。类型错误会逃逸到运行时。

建议：在 ReadonlySessionManager 上增加这两个方法，或单独暴露一个 ContextSteadySessionManager narrow interface。

建议改进 [4] checkpoint.ts:217 — 尾部 digest 无法被折叠（行为未文档化）

if (candidates.length < checkpointEveryTurns) return null;
const selected = candidates.slice(0, checkpointEveryTurns);

10 轮、checkpointEveryTurns=8：第 8 轮创建 checkpoint，第 9–10 轮 digest 永远悬空。 行为是正确的（悬空尾部由 recentDigests 窗口处理），但没有文档说明，未来维护者容易困惑。

建议在函数注释里加一句：余数 digest 不折叠，留给 recentDigests 窗口处理。

建议改进 [5] relevance.ts — 短 CJK prompt 相关性阈值过宽

return overlap >= 2 ? overlapRatio >= 0.3 : currentTokens.size <= 3 && overlapRatio >= 0.5;

CJK bigram 分词后，短 prompt（≤3 个 bigram）走 overlapRatio >= 0.5 分支， 只需一半 token 重叠就判为相关。类似"继续改"这类短 prompt 只有 1–2 个 bigram， 可能误匹配内容完全不同的 digest，导致不相关上下文被注入。

建议补充 CJK 短 prompt 的误匹配回归测试，明确当前阈值是否符合预期。

建议改进 [6] packet.ts:453 — recallLayer! 依赖隐式不变量

selectedRecall = selectedRecallItems.length > 0 ? { ...recallLayer!, items: selectedRecallItems } : undefined;

不变量成立（selectedRecallItems 来自 recallLayer?.items）， 但需追溯三层赋值才能验证。建议在循环外加一行防御性 guard 或将 recallLayer 加入 while 条件。

小问题 [7] fallback.ts — guessAction 覆盖不全

write 返回 "created" 但 write 是覆盖写，并非总是创建； ast_edit、notebook 会走到 "unknown"。 建议把常用写操作工具补进 modified 分支。

小问题 [8] session.ts:47 — preTurnLeafId 不在 branch 时静默跳过

compaction 后 branch 重建，旧 leaf ID 不存在时 digest 被静默跳过，无诊断信息。 建议加 logger.debug("preTurnLeafId not found in branch, skipping digest", ...)， 方便排查 compaction 后的 digest gap。

小问题 [9] packet.ts:430 — selectedWithinBudget 后重复 estimate

selectedWithinBudget 内部已算了 tokenEstimate，这里丢弃后重算。 建议 destructure 返回值直接复用，减少重复渲染。

小问题 [10] agent-session.ts — abort 路径三变量协议注释分散

persistedOriginalBoundary + #resetSessionStopContinuationState() + boundaryOverride 构成三方协议，相关注释分布在 L3299 和 L3420–3428 两处。 建议将不变量集中在 #generateTurnDigest 函数头注释里。

小问题 [11] settings-schema.ts — qualityWindowTokens 默认 0 不直观

默认 0 的语义是"不设质量窗口限制"，但名字暗示应填 token 数。 建议描述字段里加 "0 = no quality-window cap"。

小问题 [12] prune.ts:71 — min/max 掩盖了反向 span

const start = Math.min(fromIndex, toIndex);
const end   = Math.max(fromIndex, toIndex);

span 倒置时应产出空结果，而非静默裁剪错误范围。建议加显式断言或早期返回。

  • 替换 digest.ts JSON fallback regex 为更健壮的解析逻辑。
  • 将 SessionManagerInternal 提升到 ReadonlySessionManager 类型层，消灭四处双重 cast。
  • 补充 CJK 短 prompt 相关性回归测试，验证阈值是否符合预期。
  • 在 buildContextCheckpoint 和 #generateTurnDigest 注释里把已知不变量写清楚，降低维护成本。
  • 在 settings-schema.ts 的 qualityWindowTokens 描述里加上 "0 = no cap" 说明。
