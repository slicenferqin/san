San v0.2 · Technical Design

成熟执行闭环的代码级设计

本文把 san-v0.2-requirements-plan.html 落成实现设计：session entries、settings、 task agents、role ContextPacket、San Checks、orchestrator 状态机、测试策略和 phase 交付顺序。

设计原则

先有账本，再有自动化

任何 loop 行为都先写入可恢复的 LoopRun ledger；不要让闭环状态只存在于 prompt 或内存里。

角色视图来自 ContextPacket

v0.2 不另建上下文系统，而是在 v0.1 ContextPacket 上做 role view 裁剪和 debug refs。

Prompt workflow 可引导，不能作为终态

M1-M3 可复用 task agents 加速，但成熟交付必须由代码级 orchestrator 管状态、预算、retry 和恢复。

Supervisor 是 gate，不是 summary

Supervisor 输出必须可执行：缺陷、证据、复现、修复要求、是否 retryable、最终 verdict。

Session Entry 合同

v0.2 继续使用 CustomEntry 保存机器可读状态，避免污染 LLM context；需要进入模型的内容走 role ContextPacket 的 CustomMessageEntry，且必须可 debug。 

const SAN_LOOP_RUN_TYPE = "san.loop_run";
const SAN_LOOP_EVENT_TYPE = "san.loop_event";
const SAN_LOOP_REVIEW_TYPE = "san.review_report";
const SAN_LOOP_PACKET_TYPE = "san.loop_context_packet";

type SanLoopCustomEntry =
  | { customType: "san.loop_run"; data: LoopRunSnapshot }
  | { customType: "san.loop_event"; data: LoopEvent }
  | { customType: "san.review_report"; data: ReviewReport }
  | { customType: "san.loop_context_packet"; data: RoleContextPacketDebug };
Entry 写入时机 恢复用途
san.loop_run run 创建、状态变化、final verdict。 恢复 run header、mode、objective、当前状态、预算策略。
san.loop_event Commander 决策、Worker assignment、retry、blocked、abort。 重建 timeline 和下一步决策。
san.review_report Supervisor 每次 gate 后。 恢复缺陷列表、retry payload、最终质量判断。
san.loop_context_packet 每次 role prompt 前。 debug 角色上下文、digest refs、裁剪决策和 token 估算。

Settings

建议新增 san.executionLoop，默认关闭。v0.2 dogfood 使用 overlay 开启，发布前再决定是否暴露为稳定 CLI 模式。 

san:
  executionLoop:
    enabled: false
    defaultMode: smart
    maxRetries: 2
    maxWorkers: 3
    ledger:
      enabled: true
      persistRolePackets: true
    checks:
      enabled: true
      includeBuiltins: true
      projectDir: .omp/checks
    roles:
      commander:
        modelRole: slow
        tools: [read, search, find, task, todo]
      worker:
        modelRole: default
        tools: [read, search, find, write, edit, bash]
      supervisor:
        modelRole: advisor
        tools: [read, search, find, bash]
      oracle:
        modelRole: plan
        enabledInModes: [deep]
    budget:
      rushMaxTurns: 3
      smartMaxTurns: 8
      deepMaxTurns: 16
      reserveRatio: 0.25
默认策略 enabled 默认 false；推荐 overlay 用于 dogfood 和用户验收。当前 P4 已有代码级 transition/runner 合同、真实 task-agent executor bridge、 Commander role context 注入、/san-loop run、/san-loop stop、active-run recovery 和确定性 dogfood verifier； 预算硬上限已接入 san.executionLoop.budget.*MaxTurns，run 会在启动下一次 Commander / Worker / Supervisor 前耗尽预算时转为 blocked。 默认交互路径仍保持关闭，避免未选择用户被动进入新 loop；v0.2 以显式启用和推荐 overlay 交付验收。

角色 Agent Definitions

P2 先用 OMP task agent 体系承载角色。项目可覆盖 bundled agent，但内置版本必须能直接工作。

文件 Frontmatter 关键规则
src/prompts/agents/san-commander.md tools: read,search,find,task,todo,yield
spawns: san-worker,san-supervisor,san-oracle		 只能产出计划和派发，不直接修改代码；必须引用 run ledger。
src/prompts/agents/san-worker.md tools: read,search,find,write,edit,bash,yield
spawns: ""		 只做 assignment 范围内的实现；输出 WorkerResult。
src/prompts/agents/san-supervisor.md tools: read,search,find,bash,yield
spawns: san-oracle		 禁止 edit/write；输出 ReviewReport，verdict 必须可执行。
src/prompts/agents/san-oracle.md tools: read,search,find,yield
spawns: ""		 只给第二意见；必须声明不确定性，不改变 run 状态。

Role ContextPacket

角色上下文不是复制完整 session。P2 新增 role view builder，输入是 v0.1 ContextPacket、run ledger 和 assignment， 输出是 role-scoped prompt prelude。

角色 必须包含 必须裁剪
Commander 用户目标、全局 constraints、run ledger、Worker/Supervisor reports、recent digests。 大段文件内容、低层 shell 输出、Worker 私有 scratch。
Worker assignment、acceptance criteria、相关文件 refs、必要 checks、上一轮 defects。 无关任务 DAG、其他 Worker 细节、Oracle 争议上下文。
Supervisor plan、assignment、diff summary、tests run、checks、WorkerResult、用户验收标准。 Worker 过程性思考、无关历史 digest。
Oracle 争议问题、证据 refs、候选判断、风险。 执行细节和可由普通 Supervisor 判断的低风险事项。

Orchestrator 状态机

idle
  -> planning
  -> dispatching
  -> working
  -> reviewing
  -> passed

reviewing
  -> retrying      when verdict = needs_fix and retry budget remains
  -> blocked       when verdict = blocked or user input required
  -> failed        when retry budget exhausted
  -> passed        when verdict = pass

retrying
  -> planning      when Commander must replan
  -> dispatching   when defects map directly to WorkerAssignment

any active state
  -> aborted       on user abort
  -> blocked       on missing external dependency
恢复规则 resume 时从 session entries 重建最新 LoopRunSnapshot，如果最后状态是 active 但没有 running child process， 转为 blocked 并写入 recovery event，避免幽灵任务继续执行。

San Checks 设计

Checks 是 markdown 规则文件，不是代码插件。P3 先实现发现、解析、分组和 prompt 注入；后续可升级成 extension-backed checks。 

---
name: project-typescript-contracts
description: TypeScript implementation rules for this repository.
scope:
  paths: ["packages/coding-agent/**"]
severity: error
appliesTo: ["worker", "supervisor"]
---

- Do not use `any` unless explicitly justified.
- Do not use `ReturnType<>`.
- Prompts must live in static `.md` files.
- Run `bun check` for package-wide TypeScript validation.
阶段 行为
Discovery 读取 nearest project .omp/checks、user checks、bundled checks，按 first-wins 合并。
Planning Commander 将相关 checks 绑定到 task graph。
Worker Worker 看到 assignment 相关 checks，但不能自行关闭 gate。
Supervisor Supervisor 必须逐条给出 pass / fail / not_applicable 和证据。

Slash / SDK 入口

入口 行为 Phase
/san-loop run [--mode rush|smart|deep] <objective> 创建 execution loop run ledger，调用真实 Commander / Worker / Supervisor task agents 执行 pass / retry 闭环，并让后续真实用户 prompt 自动获得 Commander role context。 P1-P4
/san-loop status 展示当前 run ledger、active state、latest review、retry budget。 P1
/san-loop stop 写入 aborted event，停止未完成 loop，不删除历史。 P4
SDK runSanLoop() 供 dogfood、测试和未来远程控制调用；当前接受 pluggable executor，并已覆盖 pass / needs_fix retry / blocked / abort / recovery / deep Oracle gate。 P4

Phase 实施计划

P0

设计冻结

确认需求文档、技术设计、entry names、settings tree、状态机和测试矩阵。输出可以直接转 issue 的 implementation checklist。

P1

LoopRun Ledger

新增 san-loop 类型、ledger helpers、report renderer、/san-loop status。不启动三角色前，先保证状态可审计、可恢复。

P2

角色化 Task Agents

添加 bundled agent definitions、output schemas、role ContextPacket builder。先支持顺序闭环：Commander -> Worker -> Supervisor。

P3

Checks Gate 与 Retry

实现 checks discovery 和 Supervisor gate，支持 needs_fix 自动生成修复 assignment，限制 retry 次数并写入 ledger。

P4

代码级 Orchestrator

把 slash workflow 固化为 SanLoopOrchestrator / runSanLoop()：状态机、mode policy、并发 Worker、budget、abort/resume、SDK 入口。 当前已完成 transition helpers、runner 合同、pass/retry 持久化、Commander role context 注入、真实 task-agent executor bridge、/san-loop run、 /san-loop stop、hard turn budget、active-run recovery 和确定性 dogfood verifier；P4 可进入用户验收。

P5

Dogfood 与发布

先用确定性 verifier 固化 rush/smart/deep、retry、blocked、recovery、abort 合同，再用 5-8 个交付型任务或用户验收任务记录质量、失败、成本和上下文稳定性，形成发布口径。

实现清单

  1. 新增 packages/coding-agent/src/san-loop/ 并导出 types、ledger、context、checks、orchestrator。
  2. 新增 san.executionLoop settings schema 和 recommended dogfood overlay。
  3. 添加 bundled san-commander、san-worker、san-supervisor、san-oracle agent definitions。
  4. 新增 role ContextPacket debug entry，复用 v0.1 packet builder 的 digest refs 和 token estimates。
  5. 实现 .omp/checks discovery 和 bundled San checks。
  6. 新增 /san-loop registry 入口和 status/report helper。
  7. 实现 P4 transition helpers、runSanLoop() runner、Commander role context 注入和 /san-loop run 执行入口。
  8. 接入真实 task-agent executor，把 Commander / Worker / Supervisor agent 输出转换为 runner executor 结果，并持久化子 agent artifacts。
  9. 补齐 /san-loop stop、hard turn budget、active-run recovery、确定性 dogfood verifier 和 dogfood CLI script。
  10. 补齐 san-loop 测试目录，并在 changelog 的 Unreleased 记录 v0.2 execution loop 交付。
