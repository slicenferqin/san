针对代码审查发现的 cache 失效、预算失控、证据链断裂等问题的分级修复计划

Codex 批注 · 2026-07-01

本文问题清单大体成立，尤其是 P1-B、P2-C、P2-D、P2-E 和 P3-F。这些问题会影响 v0.1 的预算可信度、证据回溯和 fallback digest 质量，应该先修。

P1-A 的问题成立，但本文的方案 B 不应直接执行。 简单删除 #dropContextSteadyPacketMessagesFromActiveContext() 会让旧的全量 packet 在 active LLM context 中累积， 与 M10 的 replay 合同冲突，也会造成 digest 重复注入和 token 线性放大。P1-A 应单独设计 append-only delta packet： checkpoint 低频稳定注入，turn digest 逐轮追加 delta，避免每轮重写全量 packet。

本轮质量修复范围调整为：先实现 P1-B 总预算 gate、P2-C entryIds、 P2-D importance 0-1 契约、P2-E 多 assistant 决策提取、 P3-F digest 降级日志。P1-A append-only delta 和 P3-G/P3-H 留作后续独立设计/清理。

0. 摘要

v0.1 的 turn boundary 处理、CustomEntry/CustomMessageEntry 契约分层、never-throw 鲁棒性都做得扎实。 但有 2 个 P1 直接动摇"上下文稳态"的价值主张(cache 失效、预算失控),以及若干 P2/P3 缺陷会在 LLM digest 上线后放大。本方案按严重度 + 修复成本给出改法与测试契约。

级别问题影响修复成本
P1-A每轮 drop + 重注入 packet,active context 非 append-onlyprompt cache 从首个被删位置起全部失效,与文档核心承诺矛盾中(需选架构)
P1-Bpacket 无总量 gate,三层预算相加单条注入可达 ~15K tokens,maxTokens=2000 名不副实低
P2-CtoolEvidence.entryIds 恒为 undefined证据回溯链路断裂,digest 无法回指 journal entry极低(1 处)
P2-Dimportance 契约不一致(文档 0–1 vs normalize 1–5)LLM digest 上线即踩,0.9 被压成 1 语义全丢极低
P2-Edecisions 只扫最后一条 assistant 文本多步 turn 的中间决策全部丢失低
P3-Fcompaction 后 span 脱轨,digest 静默丢弃无日志难排查极低
P3-G每轮 O(n) 多次全 journal 扫描长会话退化低
P3-HnormalizeDigest 死代码 + checkpoint.digestCount 语义漂移latent,LLM path 上线前需清理极低

1. P1 级问题

P1-A active context 非 append-only,cache 每轮失效

agent-session.ts:6788 #buildContextSteadyPacketPrelude → agent-session.ts:3532 #dropContextSteadyPacketMessagesFromActiveContext

现状

每轮 build packet 前无条件 filter 掉 active context 里所有 packet 消息,再把当轮新 packet 通过 prependMessages 插在 user message 之前。发给模型的 token 序列因此每轮被 rewrite:

第2轮:[sys, U1, A1, P2, U2, A2]
第3轮:[sys, U1, A1, P2    U2, A2, P3, U3]  ← P2 被删,序列在第 4 个位置分叉

prefix cache 按 token 前缀匹配。第 3 轮与第 2 轮的公共前缀止于 [sys, U1, A1], 其后全部 recompute。等于每轮只能命中到首轮结束。

这恰好触发文档 context-steady-state-san.html 明确要避免的模式:
"每轮摘要 append-only … 重写旧摘要会导致大段缓存失效 … 稳定前缀尽量不变"。
journal 层确实 append-only(persist 的 entry 不删,resume 时 session-context.ts:245 过滤), 但注入 LLM 的 active context 是每轮删旧加新——正是文档反对的做法。

根因

drop 是为了防止历史 packet 在 active context 里累积(否则每轮都带 checkpoint+5 digests,token 翻倍)。 它用牺牲 cache 换取token 不膨胀。但当前设计两头都没占到:packet 放对话流末尾 + 每轮 drop, 是对 cache 最不友好的组合。

修复方案

关键洞察:packet 就紧贴在当轮 user message 前。只要不删,序列天然是 append-only 的:

第2轮:[sys, U1, A1, P2, U2, A2]
第3轮:[sys, U1, A1, P2, U2, A2, P3, U3]  ← 完整前缀命中,growing prefix
方案 A(推荐 · 目标形态)

append-only delta: 每轮只追加刚沉淀的那一条 digest 作为小消息;checkpoint 拆成独立消息,仅在 checkpoint 变化(每 8 轮)时刷新一次。

  • cache:近乎完美(只在 checkpoint 刷新那一轮失效一次)
  • 冗余:无。digest-1 只出现一次
  • 契合文档"append-only ledger + 低频 checkpoint"原意
  • 成本:中。需拆分 packet builder,digest/checkpoint 分别注入
方案 B(即时止血)

删掉 drop 调用,保留现有全量 packet builder。

  • cache:稳定(growing prefix)✅
  • 冗余:P3 重述 P2 内容,token O(N·window) 增长 ⚠️
  • 成本:极低(删 1 个方法调用 + 相关状态)
  • 定位:方案 A 落地前的过渡

建议:先上方案 B 止血(本迭代),把方案 A 列为 M5 目标。方案 A 的注入结构:

// 稳定层:仅在 latestCheckpoint.entryId 变化时重新注入,否则复用上一轮的同一条消息
[sys] → [checkpoint_message?] → [U1, A1, digest1, U2, A2, digest2, ...] → [U_now]
        ↑ 稳定前缀            ↑ append-only,每轮只 +1 条 digest

测试契约

  • 新增多轮(≥4)测试:断言第 N 轮的 mock.calls[N].context.messages 的前缀与第 N-1 轮逐条相等直到上一轮末尾(证明 growing prefix)。这是当前 M2 测试缺失的核心契约——它只跑 2 轮且只断言"packet 出现/不出现"。
  • 断言历史 packet 消息在后续轮次的位置索引不变。

注:cache 命中率本身依赖 provider,无法单测;但"token 序列 append-only"是可确定断言的可观测契约,等价保证。

P1-B packet 无总量 gate,三层预算相加

packet.ts:302 buildContextPacket · resolvePacketBudget

现状

const totalPacketTokenBudget =
    budget.packetTokenBudget          // digest,默认 2000
    + (checkpointRef?.checkpoint.tokenBudget ?? 0)   // 默认 12000,不参与 packet 级裁剪
    + (recallLayer?.tokenBudget ?? 0);               // 默认 1000
  • digest ledger 按 packetTokenBudget 裁剪(selectedWithinBudget)✅
  • recall 按自身 budget 裁剪 ✅
  • checkpoint 整块进 renderPacketContent,只在 build 时按 12000 限过,从不参与 packet 级裁剪 ❌

三层拼成一条 custom_message 注入,单条实际可达 ~12000 + 2000 + 1000 ≈ 15K tokens, 而用户配置的 contextPacket.maxTokens=2000 只约束了其中一层。packet.tokenEstimate(整块 estimate) 算出来了却从不与总预算比较后再裁。与文档"每层有预算,超预算按层裁剪"偏移。

修复方案

  1. 引入 packetTotalBudget = qualityWindowTokens>0 ? min(configuredTotal, qualityWindow−reserve) : configuredTotal。
  2. 整块渲染后,若 tokenEstimate > packetTotalBudget,按文档裁剪顺序逐层削减并重估:
    retrievedContext → oldTurnDigests(ledger 头部) → checkpoint 尾部 summary items   (preserve: currentPrompt / 最近 digest / openDecisions / modifiedFiles)
  3. 每次裁剪写入 trimDecisions(已有结构),新增 reason: "packet_total_budget"。
  4. 让 checkpoint 可裁:复用已有的 summary.nextSteps.slice(0,-1) / risks.slice(0,-1) 收缩逻辑。

测试契约

  • 构造 checkpoint + 5 digests + recall 使整块超总预算,断言最终 packet.tokenEstimate ≤ packetTotalBudget。
  • 断言裁剪顺序:recall 先于 old digests 先于 checkpoint 被削(检查 trimDecisions 的 layer 序列)。
  • 断言 currentPromptPreview 与最近 1 条 digest 永不被裁。

2. P2 级问题

P2-C toolEvidence.entryIds 恒为 undefined

fallback.ts:124 依赖 m.entryId · session.ts extractSpanMessages

现状

ToolResultMessage(ai/src/types.ts:561)没有 entryId 字段; extractSpanMessages 对 type==="message" 的 entry 直接 push entry.message,不带 entry.id。 因此 entryIds 恒为 undefined。文档承诺 digest "保留证据索引 entryId,下一轮按需重读原文"——该链路当前是断的。

修复方案

在 extractSpanMessages 里给 message shim 附上 entry.id(浅拷贝,勿改原 message 对象):

if (entry.type === "message" && entry.message !== undefined) {
-   messages.push(entry.message);
+   messages.push({ ...(entry.message as object), entryId: entry.id });
}

InlinedMessage.entryId 已是可选字段(fallback.ts:45),collectTools 直接消费,无需改 fallback。

测试契约

  • 已存在的 turn-digest.test.ts 里,构造带 toolResult entry 的 branch,断言产出 digest 的 toolEvidence[].entryIds 指向正确的 journal entry id,且该 id 可被 sessionManager.getEntry() 解析。

P2-D importance 契约不一致

normalize.ts:87 · types.ts TurnDigestMemoryCandidate · 文档示例

现状

文档与 types.ts 示例用 importance: 0.9(0–1 浮点),但 normalize 把它 round + clamp 到 1–5 整数: 

Math.max(1, Math.min(5, Math.round(obj.importance)))  // 0.9 → round → 1,语义丢失

目前 memoryCandidates 恒为空(fallback 不产出),是 latent bug;LLM digest 一上线即踩。

修复方案

统一到 0–1 浮点(与文档/types 一致,也贴合 memory 后端的相关性打分语义):

- const importance = ... ? Math.max(1, Math.min(5, Math.round(obj.importance))) : 1;
+ const importance = typeof obj.importance === "number" && Number.isFinite(obj.importance)
+     ? Math.max(0, Math.min(1, obj.importance))
+     : 0.5;  // 缺省中性值

同步:在 digest 的 prompt 模板(LLM path)里明确要求 importance ∈ [0,1]。

测试契约

  • normalizeDigest({ memoryCandidates: [{ content, type, importance: 0.9 }] }) → 断言 importance 保持 0.9。
  • 越界值(如 2、-1、NaN)→ 断言分别 clamp 到 1、0、0.5。

P2-E decisions 只扫最后一条 assistant 文本

fallback.ts:228 lastAssistantText → collectDecisions

现状

generateFallbackDigest 只对最后一个 assistant 消息抓决策。多步 turn(多次 tool call、中间多条 assistant 文本)的 中间决策全部丢失——而多步 turn 恰恰是 digest 最该捕获的。

修复方案

遍历 span 内所有 assistant 消息收集决策,总量 cap 到 MAX_ARRAY_ITEMS:

function collectAllDecisions(msgs: readonly InlinedMessage[]): string[] {
    const out: string[] = [];
    for (const m of msgs) {
        if (m.role === "assistant") out.push(...collectDecisions(msgText(m)));
        if (out.length >= MAX_ARRAY_ITEMS) break;
    }
    return truncate(out, MAX_ARRAY_ITEMS);
}
// generateFallbackDigest: decisions: collectAllDecisions(msgs)  (替代 collectDecisions(at))

测试契约

  • 构造含 2 条 assistant 文本(各带一个决策句)的 span,断言两个决策都进入 digest.decisions。

3. P3 级问题

P3-F compaction 后 span 脱轨,digest 静默丢弃

session.ts computeTurnSourceSpan / extractSpanMessages · digest.ts

现状 / 修复

若 preTurnLeafId 因 compaction/rewind 脱离当前 branch,fromEntryId 指向不存在的 id → extractSpanMessages 返回空 → generateDigest 静默 return。degrade 合理,但无日志。 在 #generateTurnDigest 的 span==null 或 spanMessages.length===0 分支加一条 logger.debug, 记录 fromEntryId / toEntryId / branchLen 便于排查。

P3-G 每轮 O(n) 多次全 journal 扫描

packet.ts collectDigestRefs · checkpoint.ts collectContextCheckpoints / coveredDigestEntryIds

现状 / 修复

collectDigestRefs、latestContextCheckpoint、coveredDigestEntryIds 每轮各自全遍历 getEntries(), 多次重复。v0.1 缓解:每轮在 prelude 构建入口只取一次 entries,单次遍历分类出 { digests, checkpoints },再传给 buildContextPacket / buildContextCheckpoint,消除重复扫描。 彻底方案(增量索引)留到 M5,当前不必过度设计。

P3-H 死代码与字段语义漂移

normalize.ts normalizeDigest · checkpoint.ts:201-210 digestCount/fromDigestEntryId

现状 / 修复

  • normalizeDigest 目前是死代码(仅测试引用,生产只走 fallback)。它是 LLM path 的预置件—— 保留,但必须先修 P2-D(importance)再随 LLM path 启用。不建议现在从 barrel 移除。
  • checkpoint.digestCount 语义漂移:entryRefs 累积增长后,digestCount: entryRefs.length 变成"累计覆盖总数"而非"本次合并数"。当前无消费方,建议正名:保留累计值改名 coveredDigestCount,另加 appendedDigestCount = appendedEntryRefs.length,避免后续误用。

4. 建议实施顺序

  1. P1-A 方案 B(止血) + P1-B(总量 gate):一次提交。这两条是"稳态"价值主张的成立前提,且都低风险。配套多轮 cache-prefix 稳定性测试 + 总预算测试。
  2. P2-C(entryIds) + P2-E(多 assistant decisions):fallback digest 质量修复,一次提交,补 turn-digest.test.ts 契约。
  3. P2-D(importance) + P3-H(死代码/命名):LLM path 上线前的清障,一次提交。
  4. P3-F(日志) + P3-G(单次遍历):可观测性与性能,可并入任意批次。
  5. P1-A 方案 A(append-only delta):列为 M5 目标,拆分 packet builder,单独评审。
验证纪律(遵循 AGENTS.md 测试规范):每条修复配可观测契约测试——运行代码断言输出/状态, 不做 source-grep,不加 expect(true) 类占位。改完跑 bun check + packages/coding-agent/test/context-steady/ 全套。 P1-A 的 cache 契约用"token 序列 append-only"这一可确定断言替代不可单测的命中率。

5. 不在本次范围

  • LLM digest 实现:M1 只有 fallback。本方案只把 importance/normalize 契约备好,不实现 LLM path。
  • qualityWindowTokens 默认值决策:当前默认 0 → 预算体系默认不生效。是否按文档设 384K 属产品决策,不在修复范围,但 P1-B 的总量 gate 会在其被启用时立即受益。
  • checkpoint 语义合并(替代机械提取):文档已承认 deterministic,留待后续。
  • enabled 默认 false → true 的开关决策:属灰度节奏,不在代码修复范围。
