San Agent Design Memo

上下文稳态：把长上下文从“能塞多少”改成“稳定工作在黄金窗口”

本文把 San 的上下文管理目标定义为 Context Steady State：每轮 agent 停下时都把上下文收敛到结构化、可裁剪、可缓存、可召回的工作态，而不是等到爆窗再被动压缩。

问题判断

现在很多模型标称 1M 甚至更长上下文，但实用质量窗口通常小于标称窗口。超过某个区间后，模型不是不能读，而是注意力分配、指令遵循、细节引用和输出稳定性都会下降。San 不应该把标称窗口当作目标，而应该维护自己的黄金工作窗口。

核心判断：长上下文的价值不是“持续堆历史”，而是给系统更多缓冲；真正进入模型主工作区的内容必须被选择、分层和稳定排布。
现象 风险 San 的策略
标称窗口越来越大 prompt 越堆越长，模型注意力被稀释 引入 qualityWindowTokens，默认按 272K-500K 工作
每轮工具调用产生大量噪声 下一轮带入原始日志，成本高且污染决策 每轮结束生成结构化 TurnDigest
摘要频繁改写前缀 破坏 prompt cache 命中率 固定前缀稳定，动态上下文放尾部
近因上下文和长期记忆混在一起 模型分不清事实、计划、偏好、证据和噪声 按层组装 ContextPacket，每层有预算和来源

目标定义

上下文稳态是一个会话不变量：一次用户 prompt 触发一轮 agent run；这一轮内部可以多次 LLM 请求、工具调用、子 agent 派工；当这一轮停止时，San 必须把这轮经历沉淀为可复用状态，并在下一轮开始前重新组装上下文。

稳定 每轮结束后上下文规模回到可控范围，不依赖模型最大窗口兜底。
可解释 下一轮模型看到的每块上下文都有来源、用途和预算，不是历史拼接。
可缓存 稳定前缀尽量不变；易变信息放后面，减少缓存失效范围。
可召回 用户历史 prompt、项目摘要、事实记忆、工具证据按需召回，不默认全带。
可裁剪 超预算时按层级、时效、相关性、证据强度裁剪，而不是简单砍旧消息。
可演进 第一版不重写 agent loop，通过 session 层和 memory 层插入策略。

工作循环

1 收到用户 prompt

解析当前意图，保留完整原文，准备生成本轮 ContextPacket。

2 组装上下文

固定前缀、历史摘要、近 5 轮、相关召回、当前 prompt 分层进入模型。

3 执行 agent run

允许多次 LLM、工具、MCP、子 agent 调用；保留原始事件日志。

4 生成 TurnDigest

本轮结束后沉淀意图、决策、文件、证据、风险、下一步。

5 进入稳态

更新摘要账本、记忆候选、缓存布局和预算索引，等待下一轮。

ContextPacket 分层

San 每轮不直接把 session history 全量交给模型，而是生成一个可审计的 ContextPacket。每层都有不同稳定性，决定它放在 prompt 的位置和缓存收益。

Stable Prefix
San persona、工具协议、开发规则、稳定 system prompt、稳定 skills 索引。
cache: high
Project State
项目级事实、仓库地图、当前分支、重要约束。只在项目状态变化后刷新。
cache: high
Checkpoint Summary
每 N 轮合并一次的稳定摘要，不每轮改写，避免破坏大块缓存。
cache: medium
Turn Digest Ledger
append-only 的轮次摘要索引。新 digest 追加在尾部，旧 digest 不重写。
cache: medium
Recent Exact Turns
最近 5 轮原文，包括用户 prompt、关键 assistant 决策和关键工具结果短摘。
cache: low
Retrieved Context
Mnemopi、CodeGraph、文件状态、MCP 资源、用户历史 prompt 的相关召回。
cache: low
Current Prompt
当前用户 prompt 完整放最后，确保最高近因权重。
attention: high
固定前缀不能塞每轮都会变化的摘要。摘要如果频繁插在最前面，缓存会被反复击穿；San 应该把稳定规则固定，把动态信息放在后段。

TurnDigest 结构

每轮结束后生成的不是普通自然语言总结，而是结构化摘要。它既服务下一轮上下文，也服务长期记忆、项目账本、调试和后续复盘。 

turnId: "2026-06-28T14:52:10.123Z-0007"
userIntent: "讨论并定义 San 的上下文稳态机制"
promptRefs:
  currentPromptHash: "sha256:..."
  relatedHistoricalPrompts: ["prompt-0002", "prompt-0005"]
actionsTaken:
  - "分析现有 agent loop 与 session 注入点"
  - "形成 ContextPacket 分层方案"
decisions:
  - "优先在 coding-agent session 层落地，不先改 core loop"
  - "每轮 agent_end 后生成 TurnDigest"
filesTouched:
  - path: "docs/research/context-steady-state-san.html"
    reason: "记录方案"
toolEvidence:
  - tool: "codegraph"
    summary: "AgentSession.prompt 是下一轮上下文注入点"

  - "Mnemopi 已支持 autoRecall / autoRetain / recallForCompaction"
openQuestions:
  - "TurnDigest 默认用 smol 还是 default 模型生成"
risks:
  - "每轮摘要会增加额外请求成本"

  - "实现 ContextSteadyState service 的最小闭环"

    type: "preference"
    importance: 0.9

关键点是：文件内容、工具输出和长日志不直接塞进摘要，只保留证据索引、短摘、路径、行号、hash 和重读策略。下一轮如果需要原文，再由 read/codegraph/memory 重新取新鲜内容。

预算策略

San 应该把模型窗口拆成三层：标称窗口、质量窗口、可用上下文预算。标称窗口越大，San 的缓冲越多，但默认不把工作上下文推满。

参数 建议默认 说明
model.contextWindow 来自模型 catalog 模型标称上限，只作为硬上限和应急缓冲。
san.contextSteady.qualityWindowTokens 384000 San 主工作窗口。1M 模型也默认控制在这个区间。
reserveRatio 0.20 给输出、工具结果、模型偏差和临时膨胀预留。
recentTurns 5 近因原文保留窗口，不和长期摘要混淆。
digestMaxTokens 2000 单轮 digest 上限。普通轮次可更小，复杂轮次可接近上限。
checkpointEveryTurns 8 每 8 轮把 digest ledger 合并成稳定 checkpoint。 
hardWindow = model.contextWindow
qualityWindow = min(hardWindow, san.contextSteady.qualityWindowTokens)
reserve = max(qualityWindow * reserveRatio, 24000)
usableBudget = qualityWindow - reserve

缓存命中设计

Prompt cache 的重点不是“尽量短”，而是“前缀稳定”。San 的上下文组装必须减少前缀抖动。

规则 原因 落地方式
稳定内容放前面 缓存从前缀匹配开始，越前面的稳定内容收益越高。 system prompt、工具协议、稳定 skills、项目规则固定排序。
每轮摘要 append-only 重写旧摘要会导致大段缓存失效。 旧 TurnDigest 不改，只追加新 digest。
checkpoint 低频改写 合并摘要不可避免会改写，但可以降低频率。 每 8 轮或超预算时合并一次。
动态召回靠后 召回结果每轮变化大，放前面会污染缓存。 Mnemopi、CodeGraph、文件状态放在 current prompt 前。
当前 prompt 最后 近因权重最高，减少模型忽略当前任务的概率。 用户原文不摘要、不截断，除非超过硬限制。

与现有 San 代码的接入点

第一版不要重写 packages/agent 的 core loop。更稳的做法是在 packages/coding-agent 的 session 层实现上下文策略。

目标 入口文件 建议改法
下一轮 prompt 前注入 ContextPacket packages/coding-agent/src/session/agent-session.ts 在 AgentSession.prompt() 进入 #promptWithMessage 前组装 prelude messages。
每轮结束生成 TurnDigest packages/coding-agent/src/session/agent-session.ts 监听 agent_end，调用 digest summarizer，写入 session entry 或 San ledger。
长期记忆召回 packages/coding-agent/src/mnemopi/state.ts 复用 recallForContext、maybeRetainOnAgentEnd，增加 San context tags。
应急压缩 packages/coding-agent/src/session/agent-session.ts 继续使用现有 compact()，但定位为超预算兜底，不是主流程。
Compaction 前补充记忆上下文 session_before_compact hook 把相关 digest、memory recall、项目事实作为 extra context 给摘要器。
子 agent 上下文隔离 packages/coding-agent/src/task/executor.ts 给 subagent 传入子任务专属 ContextPacket，不继承父会话全量历史。

配置命名建议先放在 san.contextSteady 下，不直接复用现有 compaction。两者语义不同：context steady 是常态工作流，compaction 是应急或手动压缩。

MVP 分阶段

阶段 交付物 验证方式
Phase 1 实现 TurnDigest 生成和持久化；不改变下一轮上下文。 跑一轮 San，确认 agent_end 后有 digest entry。
Phase 2 实现 ContextPacket 组装，只注入 digest ledger + 近 5 轮。 打印调试视图，核对 prompt 层级和 token 预算。
Phase 3 接入 Mnemopi recall 和历史 prompt recall。 构造跨会话问题，确认能召回相关偏好和决策。
Phase 4 实现 checkpoint 合并和 append-only 缓存布局。 比较多轮后 provider prompt cache 命中和上下文规模。
Phase 5 给 subagent 传递任务专属 ContextPacket。 验证 worker 不继承无关父历史，但能拿到任务所需事实。

风险与边界

摘要成本 每轮摘要会增加请求。需要用 smol role、异步摘要、低优先级队列控制成本。
摘要失真 Digest 不能替代证据。关键文件、工具输出必须保留可重读引用。
缓存与质量冲突 为了缓存不能牺牲当前 prompt 的近因权重；动态上下文应靠后但仍在当前 prompt 前。
过度自动记忆 不是所有信息都应该进入长期记忆。需要 memoryCandidates 和 retain policy。
调试复杂度 上下文被组装后，必须有可查看的 ContextPacket debug view。
早期不要重写 loop core agent loop 是高风险区。第一版用 session 层策略验证价值。

结论

San 的上下文稳态可以成为它区别于普通 agent harness 的核心能力：不是让模型背负越来越长的历史，而是持续维护一个可用、可控、可缓存的工作记忆层。

最小落地路径是：每轮 agent_end 生成 TurnDigest，下一轮 prompt() 前组装 ContextPacket，默认控制在 qualityWindowTokens 内；现有 compaction 保留为兜底，Mnemopi 作为长期记忆和相关召回。

这条路线足够低侵入，也足够接近最终形态。先跑出稳定闭环，再逐步增强策略、缓存命中和 subagent 隔离。
