San v0.2 · Requirements Plan

成熟执行闭环，而不是最小三角色 Demo

v0.2 的目标应从“能跑一遍 Commander / Worker / Supervisor”上调为“可交付的成熟执行闭环”： 它必须能持续接手复杂开发任务，拆解、执行、验收、修复、复盘，并把闭环状态沉淀到 v0.1 的 Context Steady State 之上。

版本定位

v0.1 已解决上下文稳态问题：每轮工作可以被结构化沉淀、裁剪、复用。v0.2 应解决执行质量问题： San 不只是“连续调用工具的单 agent”，而是一个能管理任务生命周期的闭环系统。

v0.2 定义 San v0.2 = Mature Execution Loop。核心交付是一个可用于真实开发任务的 Commander / Worker / Supervisor 闭环，带结构化计划、权限隔离、检查门禁、失败重试、预算控制、可审计 run ledger 和 dogfood 验收。

不做 原因 后续版本

成熟闭环标准

任务状态可控

每个 run 有明确目标、范围、任务 DAG、当前状态、阻塞项、预算和最终 verdict。

角色职责硬隔离

Commander 不直接改代码，Worker 不无限派发，Supervisor 不写代码；权限边界由工具集和 prompt 双重限制。

验收不是口头总结

Supervisor 必须基于 diff、测试输出、静态检查、用户验收标准和项目规则给出结构化报告。

失败能闭环

缺陷必须分类为 retryable / blocked / out-of-scope，并生成下一轮修复任务，而不是让主 agent 自行猜测。

上下文受治理

Commander、Worker、Supervisor 都消费 ContextPacket，但每个角色获得裁剪后的角色视图。

运行可审计

run ledger 保存计划、子任务、worker 结果、supervisor 报告、测试证据、预算、重试和最终交付。

角色需求

角色 职责 工具边界 输出合同
Commander 理解目标、拆解任务 DAG、选择执行模式、分配 Worker、读取报告、决定 retry / stop / escalate。 读、搜索、任务派发、todo、run ledger；默认不写文件、不跑高风险命令。 LoopPlan、WorkerAssignment、LoopDecision。
Worker 执行边界明确的子任务，提交 diff、运行局部验证，说明假设和残留风险。 读、编辑、写入、shell、测试；默认不能创建新子任务或改 run policy。 WorkerResult，包含 touched files、commands、verification、risks。
Supervisor 审查计划和结果，跑/建议检查，识别缺陷、回归风险、缺失测试和验收缺口。 读、搜索、shell 验证；禁止 edit/write。 ReviewReport，包含 verdict、defects、required fixes、confidence。
Oracle 只在 deep 模式、Supervisor 不确定、或关键架构争议时给第二意见。 只读上下文和报告；不直接改代码、不派发任务。 OracleOpinion，必须引用证据和不确定性。

执行模式

模式 适用任务 闭环强度 模型/预算策略
rush 小修、小范围变更、明确 bug。 1 Commander + 1 Worker + lightweight Supervisor。 低预算，最多一次 retry。
smart 默认开发任务，跨 2-6 个文件，有测试风险。 完整计划、Worker 执行、Supervisor gate、必要 retry。 中预算，最多两轮 retry。
deep 架构、跨模块重构、发布前质量门禁。 多 Worker 可并行探索，Supervisor 可调用 Oracle。 高预算，要求更完整证据和复盘。

San Checks

v0.2 的成熟度关键在 checks。Checks 不是普通提示词，而是可复用的验收规则，可以来自仓库、团队或用户配置。

Check 类型 示例 Supervisor 如何使用
Project Convention 不使用 any、不使用 ReturnType<>、prompts 必须在 .md。 检查 diff 和实现方案是否违反项目规则。
Behavior Contract 用户可见输出、错误映射、状态转换、CLI 参数行为。 要求 Worker 补足合同级测试或解释为何不需要。
Safety Gate 危险命令、权限升级、外部写操作、敏感文件。 阻止或要求用户确认，不让 Worker 自行越权。
UX / TUI 渲染 sanitization、路径缩短、预览截断、streaming preview。 对 UI/TUI 变更跑专门 review prompt。

里程碑

M1

LoopRun Ledger 与结构化合同

新增 run ledger、LoopPlan、WorkerResult、ReviewReport、LoopDecision 类型和持久化。验收标准是单 agent workflow 也能完整记录一次 run 的状态、证据和 verdict。

M2

三角色 Task Agents 与角色 ContextPacket

实现 Commander / Worker / Supervisor agent 定义、工具边界和角色视图裁剪。验收标准是同一任务可由三角色顺序完成，且每个角色只看到所需上下文。

M3

San Checks 与 Supervisor Gate

实现 .omp/checks 规范、内置项目检查、Supervisor 报告聚合、needs_fix retry。验收标准是故意引入缺陷时 gate 能稳定拦截并生成修复任务。

M4

成熟 Orchestrator

从命令/skill workflow 升级到代码级 orchestrator：状态机、预算、并发 Worker、retry policy、abort/resume、run report。验收标准是真实复杂任务能自动闭环到 pass 或 blocked。

M5

Dogfood 与发布门槛

用确定性闭环 verifier 和 5-8 个交付型任务验证 rush/smart/deep，形成 v0.2 dogfood 报告。验收标准是闭环合同稳定、质量门禁有效、失败能收敛到 pass / blocked / aborted，且不会污染上下文或无限循环。

Phase 拆分

v0.2 的 phase 不按“先做一个最小 demo”拆，而按成熟闭环的依赖链拆。每个 phase 都必须留下可 dogfood 的端到端能力，同时给下一 phase 提供稳定合同。

Phase 主题 交付 验收
P0 设计冻结 需求基线、技术设计、状态机、entry schema、settings 和测试矩阵。 能从文档直接拆出 M1-M5 issue，不再争论 v0.2 是否只是 prompt workflow。
P1 Run Ledger 底座 san.loop_run、san.loop_event、san.review_report custom entries，run report renderer。 不用三角色也能记录一次完整 run，resume 后状态可恢复。
P2 角色化执行 Commander / Worker / Supervisor agent definitions、角色 ContextPacket、权限边界、结构化输出。 同一真实任务可按三角色顺序完成，且 Worker/Supervisor 的可见上下文和工具集不同。
P3 Checks Gate .omp/checks 规范、内置 San checks、Supervisor gate、needs_fix retry payload。 故意引入项目规则违规或缺测试变更时，Supervisor 能稳定拦截并生成修复任务。
P4 代码级 Orchestrator loop 状态机、mode policy、预算上限、并发 Worker、retry policy、abort/resume。 复杂任务可自动推进到 pass / blocked / failed，不依赖主 agent 记住流程。
P5 质量收敛 dogfood 报告、回归测试、失败样本复盘、默认配置建议、发布说明。 6 个交付型验收场景覆盖 rush/smart/deep、retry、Oracle、预算、恢复和 abort；真实模型任务作为用户验收阶段继续抽样。
Phase 约束 P1-P3 可以通过 workflow 和 task agents 加速，但 P4 必须落到代码级 orchestrator。否则 v0.2 只有“能演示的闭环”，没有可恢复、可审计、可限制预算的成熟闭环。

验收标准

  • 能用一个命令或模式启动成熟闭环，例如 /san-loop 或等价 CLI/SDK 入口。
  • 每个 run 都有可查看的 run ledger，包含计划、任务、执行结果、审查、重试和最终 verdict。
  • Commander / Worker / Supervisor 权限边界可测试，越权行为会被拒绝或降级。
  • Supervisor 不只总结，而是基于 diff、测试和 checks 产出可执行缺陷列表。
  • 至少支持 needs_fix 自动回到 Commander 生成修复任务，并限制最大 retry 次数。
  • 支持 rush/smart/deep 三种闭环强度，且预算和模型选择有明确策略。
  • ContextPacket 被角色化消费，run 多轮后不破坏 v0.1 的上下文稳态。
  • dogfood 验收覆盖 bug fix、跨文件功能、测试补强、CLI 行为、预算耗尽、恢复和用户中止；真实模型任务在用户验收阶段继续抽样。

实现取舍

建议路线 M1-M3 可以先利用现有 task agents、skills、hooks、ContextPacket 和 session custom entries 快速成型； 但 v0.2 不能停在 prompt workflow。M4 必须收敛到代码级 orchestrator，否则“成熟闭环”的状态、预算、retry 和 resume 都不够可靠。
方案 优点 缺点 v0.2 判断
纯 slash command / prompt workflow 最快，可 dogfood。 状态机弱，难做预算和恢复。 只适合作 M1-M2 引导，不是最终交付。
Skill + task agents 复用现有能力，职责隔离清晰。 跨轮状态和重试仍需宿主协调。 作为角色定义和 checks 分发层。
代码级 orchestrator 状态、预算、并发、retry、resume 都可控。 实现成本高，需要更强测试。 v0.2 必须完成，否则不算成熟闭环。

推荐下一步

  1. 把本文作为 v0.2 需求基线，先审一轮范围：成熟闭环、GBrain 后移、memory 保持候选态。
  2. 使用 san-v0.2-technical-design.html 作为实现设计，落到具体模块、entry 类型、settings、agent definitions 和测试文件。
  3. 先实现 M1 LoopRun ledger，避免三角色 workflow 先跑起来但没有可审计状态。
  4. M2/M3 已用 role ContextPacket、San Checks、agent definitions 和 focused tests 验证角色边界与 gate 合同。
  5. M4 已完成代码级 orchestrator、真实 task-agent bridge、预算硬上限、role context 注入、run/stop/status 入口和 active-run recovery。
  6. M5 已用确定性 verifier 固化 rush pass、smart retry、deep Oracle blocked、hard budget、active-run recovery、operator abort 合同；交付后用真实模型任务继续抽样即可。
