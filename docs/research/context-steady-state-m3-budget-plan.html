Context Steady State · M3

预算稳态是 v0.1 的闭环，不是再加一层摘要

M1 建立 TurnDigest 账本，M2 证明下一轮能消费 ContextPacket。M3 的目标是证明多轮后上下文不会线性膨胀： 每层有预算、裁剪可解释、debug entry 能对照本地估算与真实请求规模。

范围

做 不做
增加 qualityWindowTokens 和 reserveRatio，把 ContextPacket 可用预算从配置推导出来。 不接 Mnemopi recall，不做 GBrain，不做 checkpoint rewrite。
给 turn_digest_ledger 写 layer budget、token estimate、裁剪原因和保留 refs。 不改变 TurnDigest schema 的核心含义，不把 digest 写入长期 memory。
在 san.context_packet debug entry 中记录 context budget snapshot，方便真实会话验收。 不重写 compaction 策略；compaction 仍是兜底，不是 M3 主逻辑。

配置

"san.contextSteady.qualityWindowTokens": 0
"san.contextSteady.reserveRatio": 0.2
"san.contextSteady.contextPacket.maxTokens": 2000

qualityWindowTokens = 0 表示不强行覆盖模型窗口；M3 先只把它作为 debug/预算推导输入。 有效 ContextPacket 预算取 min(contextPacket.maxTokens, qualityWindowTokens * (1 - reserveRatio))。 如果 quality window 为 0，则沿用 contextPacket.maxTokens。

Debug Payload

{
  "tokenEstimate": 1800,
  "tokenBudget": 2000,
  "budget": {
    "qualityWindowTokens": 12000,
    "reserveRatio": 0.2,
    "reservedTokens": 2400,
    "packetTokenBudget": 2000,
    "configuredPacketMaxTokens": 2000
  },
  "layers": [
    {
      "name": "turn_digest_ledger",
      "entryRefs": ["..."],
      "tokenEstimate": 1800,
      "tokenBudget": 2000,
      "trimmed": 3
    }
  ],
  "trimDecisions": [
    { "layer": "turn_digest_ledger", "reason": "token_budget", "omitted": 3 }
  ]
}

这份 debug entry 是验收依据：它需要回答本轮为什么保留这些 digest、为什么裁掉那些 digest、最终注入大约占多少 token。

验收

  1. 构造 8-12 轮会话，每轮写出 san.turn_digest。
  2. 每个真实用户轮前写出 san.context_packet 和隐藏 san.context_packet.injected。
  3. 当 digest ledger 超过预算时，trimDecisions 包含 token_budget 或 recent_limit。
  4. 最终 packet 的 tokenEstimate 小于等于 tokenBudget。
  5. ContextPacket 注入消息不在 active context 或 resumed session 中重复累积。
M3 完成后，San v0.1 才能说“上下文稳定收敛”。M4 的 recall 可以增强相关性，但不能替代 M3 的预算闭环。

踩坑

不要把模型最大窗口当工作目标。500K 或 1M 只是硬上限，San 的目标是质量窗口内的可解释上下文。
不要只测 builder helper。至少要有 AgentSession 集成测试，验证 session entry、下一轮 payload 和恢复路径都符合预算语义。
